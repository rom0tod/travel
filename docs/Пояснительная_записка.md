## 1. Введение

### Идея проекта
Большинство планировщиков путешествий, доступных школьнику и
студенту, либо платные, либо привязаны к экосистеме (Google Maps,
Maps.me, Booking). Хочется отдельный «личный» инструмент:

- собрать поездку из точек, разложенных по дням;
- видеть маршрут на карте;
- следить за бюджетом;
- легко делиться готовым маршрутом с друзьями.

TripPlanner закрывает именно эту нишу.

### Какие задачи решает
1. Снимает с пользователя необходимость держать маршрут «в голове»
   или в плоской заметке.
2. Даёт визуальное представление поездки (карта Leaflet,
   accordion по дням).
3. Помогает не выйти за бюджет и видеть распределение расходов
   по категориям.
4. Превращает приватный маршрут в публикацию для других
   путешественников: лайки, комментарии, поиск.

## 2. Архитектура и проектирование

### 2.1. Принципы
- **Слойная архитектура.** Код разделён на изолированные
  ответственности: `data/` — модели и сессия БД, `forms/` —
  валидация ввода, `api/` — REST-ресурсы, `tools/` — утилиты,
  `templates/` + `static/` — представление.
- **Фабрика приложения.** `create_app()` в [main.py](../main.py)
  позволяет создавать конфигурации под dev/prod и упрощает
  тестирование.
- **Слой данных не знает о Flask.** Модели можно переиспользовать
  в скриптах сидера или CLI без поднятия HTTP-сервера.
- **Конфигурация в одном месте.** Все лимиты, пути и константы —
  в [config.py](../config.py) и завязаны на переменные окружения,
  что критично для деплоя.

### 2.2. ORM-модель данных
Семь сущностей с естественными связями:

```
User ───< Trip ───< Place
       │       └─< Expense
       │       └─< Comment >─── User
       │       └─< Like    >─── User
       │       └─>< Tag   (many-to-many)
       └────────< Comment
       └────────< Like
```

Все модели наследуются от `SqlAlchemyBase` и подмешивают
`SerializerMixin` для единого `to_dict()` в REST API.

### 2.3. Сценарий «создать поездку с точкой на карте»

1. Пользователь заполняет форму поездки (`forms/trip.py`,
   `WTForms` + CSRF от Flask-WTF).
2. View-функция `create_trip` сохраняет объект в БД через
   сессию SQLAlchemy.
3. На странице деталей пользователь нажимает «Добавить точку».
4. На странице точки JS-функция `initPlacePicker`
   ([static/js/map.js](../static/js/map.js)) поднимает карту
   Leaflet. Клик по карте обновляет hidden-инпуты с координатами.
5. Альтернативно пользователь вводит адрес, JS делает
   `GET /api/geocode?address=…`. Серверный эндпоинт обращается
   к Nominatim, возвращает JSON с координатами.
6. Сохранение точки приводит к обновлению карты и расписания.

### 2.4. Безопасность

| Угроза | Защита |
| --- | --- |
| Подсмотренный пароль в БД | Хэширование `werkzeug.security` |
| CSRF на POST-формах | Токен `flask_wtf.FlaskForm` |
| Просмотр чужой приватной поездки | Метод `Trip.is_visible_to(user)` |
| Изменение чужой поездки | `Trip.is_editable_by(user)` + проверка во view и API |
| Загрузка опасного файла | Проверка расширения + `secure_filename`, лимит 8 МБ |
| Перебор пароля | Сообщение об ошибке не уточняет, что именно неверно |

### 2.5. Внешние интеграции

- **OpenStreetMap (тайлы)** — без ключа, через CDN.
- **Nominatim** — поиск координат по адресу, обязательно с
  собственным User-Agent (требование сервиса).

## 3. Применённые технологии

| Категория | Технологии и реализация |
| --- | --- |
| Управление зависимостями | `requirements.txt` |
| Веб-фреймворк | Flask с фабрикой приложения |
| ORM | SQLAlchemy 2.0, declarative_base через `DeclarativeBase` |
| Сериализация моделей | `sqlalchemy_serializer.SerializerMixin` |
| Аутентификация | Flask-Login, хэш паролей, remember-me |
| Формы | Flask-WTF + WTForms, кастомные валидаторы |
| Файлы | загрузка обложек и аватаров, отдача через `/static/uploads/` |
| REST API | Flask-RESTful, 8 ресурсов CRUD |
| Сторонний API | Nominatim (геокодирование адресов) |
| Шаблоны | Jinja2 с наследованием, фильтры (`ru_date`, `ru_datetime`), контекст-процессор |
| Frontend | Bootstrap 5, Bootstrap Icons, Leaflet 1.9, AJAX через `fetch` |
| Хранение | SQLite (dev), PostgreSQL (prod) |
| Хостинг | Render.com (Blueprint через `render.yaml`) + альтернатива PythonAnywhere |
| Production WSGI | `gunicorn` |

## 4. Что нового по сравнению с типовыми проектами курса

- **Интерактивная карта Leaflet** с маршрутами по дням и
  выбором координат кликом.
- **Геокодер на стороннем API** с обработкой ошибок сети.
- **Многоуровневое разделение прав:** публичная/приватная поездка,
  владелец может удалить чужой комментарий под своей публикацией.
- **Постатейный учёт бюджета** с сводкой по категориям.
- **Чистая REST-обёртка**: одинаковый формат ответа, явные коды
  состояния, поддержка пагинации.
- **Production-ready деплой**: Procfile + render.yaml + wsgi.py +
  чтение конфигурации из переменных окружения.

## 5. Тестирование

Тестирование проводилось вручную по сценариям:

1. Регистрация → создание поездки → добавление точки → проверка
   на карте → расход → лайк → комментарий другим пользователем.
2. API-проверки через `curl`:
   - `GET /api/trips` — список публичных поездок.
   - `POST /api/trips` без авторизации — `403`.
   - `GET /api/trips/<id>` для приватной чужой поездки — `403`.
   - `GET /api/geocode?address=Москва` — корректные координаты.
3. Граничные случаи: пустой бюджет, точка без координат, поездка
   на 1 день, поездка на 300+ дней.
4. Безопасность: попытка отправить форму без CSRF-токена → 400.
