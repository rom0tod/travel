# Техническое задание

## Проект «TripPlanner»

**Авторы:** Роман Тодор

**Модуль:** WebServer + REST API
**Тип:** Веб-приложение (Flask)

---

## 1. Назначение

TripPlanner — социальный планировщик путешествий, позволяющий
пользователю собрать маршрут поездки по дням с привязкой к карте,
вести бюджет и делиться готовыми поездками с сообществом.

## 2. Целевая аудитория

- Студенты и школьники, планирующие самостоятельные поездки.
- Путешественники, которые хотят сохранить и переиспользовать
  свои маршруты, а также вдохновляться чужими.

## 3. Функциональные требования

### 3.1. Учётные записи
- Регистрация по логину и e-mail с проверкой уникальности.
- Хэширование паролей (Werkzeug).
- Вход с опцией «запомнить меня».
- Публичный профиль пользователя с аватаром и текстовым описанием.

### 3.2. Поездки
- Создание / редактирование / удаление поездки.
- Поля: название, направление, описание, даты, бюджет, обложка,
  теги, флаг публичности.
- Просмотр поездки доступен владельцу всегда, остальным — только
  если поездка публичная.

### 3.3. Маршрут
- Добавление точек интереса с координатами (широта/долгота).
- Автогеокодирование адреса через сторонний API
  Nominatim (OpenStreetMap).
- Распределение точек по дням поездки.
- Шесть категорий точек: достопримечательность, жильё, еда,
  транспорт, активность, прочее.
- Отображение всех точек на интерактивной карте Leaflet.

### 3.4. Бюджет
- Учёт расходов по 6 категориям и в произвольной валюте.
- Автоматический подсчёт суммы трат и остатка от бюджета.
- Индикатор перерасхода.

### 3.5. Социальные функции
- Лайк публичной поездки (AJAX без перезагрузки).
- Комментарии: оставить, удалить (автор или владелец).
- Explore-страница: поиск по тексту, фильтр по тегам, пагинация.

### 3.6. REST API
- Полный CRUD поездок: `GET/POST /api/trips`,
  `GET/PUT/DELETE /api/trips/<id>`.
- CRUD точек маршрута: `/api/trips/<id>/places[/<pid>]`.
- Расходы: `/api/trips/<id>/expenses[/<eid>]`.
- Пользователи (read-only): `/api/users`, `/api/users/<id>`.
- Прокси-эндпоинт геокодера: `/api/geocode?address=...`.
- Формат ответа: `{"success": true, ...}` или
  `{"success": false, "error": "..."}`.

## 4. Нефункциональные требования

| Свойство | Требование |
| --- | --- |
| Время отклика | < 300 мс на типовой запрос на локальной БД |

## 6. Структура хранилища

7 ORM-моделей:
- `User` — учётные записи.
- `Trip` — поездка (FK → User).
- `Place` — точка маршрута (FK → Trip).
- `Expense` — расход (FK → Trip).
- `Comment` — комментарий (FK → Trip, FK → User).
- `Like` — связь «лайк» (FK → Trip, FK → User; UNIQUE).
- `Tag` — тег (many-to-many → Trip через `trip_tags`).

## 7. Внешние интерфейсы

- **Веб-страницы** — рендеринг Jinja2.
- **REST API** — формат JSON.
- **Геокодер Nominatim** — `GET https://nominatim.openstreetmap.org/search`
  с обязательным User-Agent.

## 9. Критерии приёмки

- Все маршруты возвращают HTTP 200/302 для авторизованного пользователя.
- API возвращает корректный JSON по всем `GET`-эндпоинтам без авторизации.
- При попытке доступа к чужой приватной поездке возвращается 403.
- Геокодер возвращает координаты для адреса «Москва».
- Демо-сидер создаёт минимум 3 поездки, 6 точек, 3 пользователей.
